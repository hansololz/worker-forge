# Default theme

Every worker with a GUI ships with the same default theme unless the user asked for something else during the interview. The goal is a worker that looks at home next to Discord, Slack, Linear, or any other modern desktop app — not the bare-Tkinter look that screams "Python script with a window stapled on."

If the user picked Light or USER_PROVIDE for the color theme, skip this file and theme accordingly. But default is dark, and dark means *this* dark.

## Why a default at all

Worker authors aren't UI designers. Half the GUI workers that come out of this skill would otherwise ship with system-default chrome — a white title bar on a dark canvas, square 1990s-era buttons, the wrong font, hairline borders that look broken on HiDPI. That look isn't neutral; it makes the worker feel like a debug tool. A single default theme that's *good* means every worker the user builds looks intentional from the first run.

The default below is calibrated against Discord's dark mode because that's the reference users have in their heads when they say "modern dark theme." If you find yourself fighting it, fight it — the goal is "looks good," not "exactly matches these hex codes."

## Palette

Discord-dark adjacent. Stick to these unless the user asked for a custom palette.

| Token | Hex | Use for |
|-------|-----|---------|
| `bg.base`        | `#1E1F22` | Window background, the largest surface |
| `bg.surface`     | `#2B2D31` | Cards, panels, the main content area |
| `bg.elevated`    | `#313338` | Inputs, dropdowns, anything that visually floats above `surface` |
| `bg.hover`       | `#3A3C42` | Hover state on buttons, list rows, menu items |
| `bg.active`      | `#404249` | Pressed / selected state |
| `border.subtle`  | `#3F4147` | Hairline dividers between panels |
| `text.primary`   | `#F2F3F5` | Headings, body text |
| `text.secondary` | `#B5BAC1` | Labels, descriptions, secondary copy |
| `text.muted`     | `#80848E` | Placeholder text, disabled state, timestamps |
| `accent`         | `#5865F2` | Primary buttons, focus rings, active tab underline |
| `accent.hover`   | `#4752C4` | Accent on hover |
| `success`        | `#23A55A` | "Done", "Connected", positive status pills |
| `warning`        | `#F0B232` | "Action required" without being an error |
| `danger`         | `#F23F42` | Error text, destructive button |

Two things to notice. First, the background isn't pure black — pure black on an OLED-ish display reads as a void, and pure black plus rounded corners looks gothic instead of crisp. Use `#1E1F22`. Second, there's a *progression* from `bg.base` to `bg.surface` to `bg.elevated`: each one is a notch lighter, which is what makes panels look like they're sitting on top of the window instead of cut into it.

## Corners

Round them. Square corners are the single biggest tell that a GUI was thrown together quickly.

- **Window corners**: leave to the OS — the OS already rounds them on macOS and Windows 11, and on Linux it depends on the window manager. Don't try to draw your own.
- **Cards / panels / surfaces**: 12 px radius.
- **Buttons**: 8 px radius.
- **Inputs (text fields, dropdowns)**: 6 px radius.
- **Pills / status badges**: fully rounded (`border-radius: 9999px` in CSS-land, or `radius = height / 2` in code).

The progression matters. Bigger surfaces get bigger radii; small interactive bits get smaller ones. Uniform corners on everything looks toy-like.

## The top-bar rule

**The top bar must match the body color. Never leave it the default white (or default any-OS-chrome) when the body is dark.** This is the single most common visual bug in workers generated by this skill, and it makes an otherwise-fine worker look unfinished.

What "top bar" means depends on the framework. In each case, the fix is to draw the title bar yourself or theme it to match the body:

- **Tkinter**: by default, the window title bar is the OS chrome — light on light-mode OSes, dark on dark-mode OSes, never matching `bg.base`. Options: (a) call `root.overrideredirect(True)` and draw your own title bar as the first row in the window using `bg.base` for the background, with a label for the title and a close button. (b) on Windows 10+, use `pywinstyles` or set `DWMWA_USE_IMMERSIVE_DARK_MODE` via `ctypes` so the native title bar follows the app's dark theme. (a) is the more reliable fix; (b) is less code if you only target Windows.
- **PySide6 / PyQt**: set `Qt.FramelessWindowHint` and draw a custom title bar widget at the top of the central widget, styled with `bg.base`. Add window-drag handling on the custom bar so the window remains movable.
- **Electron / Tauri**: set `titleBarStyle: 'hidden'` (Electron) or `decorations: false` (Tauri) and render the title bar in the renderer process with the app's palette. Add the platform-specific window-control buttons or use a library like `custom-electron-titlebar`.
- **Native (SwiftUI on macOS, WinUI on Windows)**: prefer the platform's title-bar-tint API rather than going frameless. SwiftUI: `.toolbarBackground(Color(hex: "#1E1F22"), for: .windowToolbar)`. WinUI: `AppWindow.TitleBar.BackgroundColor = ...` plus matching `ButtonBackgroundColor`.

Whichever path you take, the title bar's background must be exactly `bg.base` (or one shade darker, if you want a subtle separator) — *not* white, *not* the OS default, *not* a slightly-different-shade-of-dark that catches the eye.

## Typography

- Sans-serif system stack. Don't ship a custom font unless the user asked for one.
  - macOS: `-apple-system, "SF Pro Text"`
  - Windows: `"Segoe UI Variable Text", "Segoe UI"`
  - Linux: `"Inter", "Cantarell", "Ubuntu"`
- Body: 14 px, regular weight, `text.primary`, line-height 1.5.
- Labels / secondary: 13 px, `text.secondary`.
- Headings: 16–22 px, semibold, `text.primary`.
- Monospace where it's actually code/paths: `"JetBrains Mono"` if available, else `"SF Mono", "Cascadia Mono", monospace`.

Avoid: shadowed text, all-caps headings, anything italic-by-default.

## Spacing

- Page padding: 16 px on small windows, 24 px on anything larger.
- Vertical rhythm between sections: 16 px.
- Inside a card: 12–16 px padding.
- Button padding: 8 px vertical, 16 px horizontal.

The rule of thumb is "more space than you think." Cramped UIs read as toy UIs.

## Components

A few specific bits the workers will almost always need:

### Buttons

Primary: filled, `accent` background, `text.primary` text, 8 px radius. Hover: `accent.hover`. No drop shadow.

Secondary: `bg.elevated` background, `text.primary` text, 1 px solid `border.subtle` border. Hover: `bg.hover`.

Destructive: `danger` text on `bg.elevated` background; hover adds a faint `danger`-tinted overlay (not a fully-red button — that's too loud for a worker).

### Inputs

`bg.elevated` background, 1 px `border.subtle` border, 6 px radius, 8 px vertical padding, 12 px horizontal. Focus state: border becomes `accent`. No outer glow.

### Status / activity

When the worker is doing something, show it. A 2-px-tall progress bar at the top of the content area, full width, `accent` color, with the indeterminate animation when there's no progress to report. Workers that just freeze with no feedback are the workers users distrust.

### Empty states

When there's nothing to show (no items yet, no results), don't leave a blank panel. Two lines of `text.secondary`-colored copy in the center: a one-liner about what would normally be here, and a one-liner about how to make something happen. This is the difference between a worker that feels broken on first run and one that feels considered.

## Applying it

How you actually wire this into a framework depends on what the user picked in the interview:

- **Tkinter** — use `ttk.Style().configure(...)` for the widget styles, and set background colors directly on `tk.Frame` / `tk.Toplevel`. Drop the Tk default theme (`style.theme_use('clam')` first — `clam` is the most themeable of the built-ins). Custom title bar via `overrideredirect(True)` as described above.
- **PySide6 / PyQt** — write a QSS stylesheet using the palette tokens. Set it on the `QApplication` so it cascades. Use `Qt.FramelessWindowHint` plus a custom `QWidget` title bar.
- **Electron** — a single CSS file with `:root` custom properties for the palette tokens, then class-level styles. `titleBarStyle: 'hidden'` and a `<div class="titlebar">` at the top of the body. Tailwind is fine if the user already has it; not worth pulling in just for this.
- **Tauri** — same CSS approach as Electron. `decorations: false` in `tauri.conf.json`.

Whichever framework, the palette tokens come from this file. Don't redefine the colors in the worker's source — paste this palette into the worker's `<os>/resources/theme.{css,json,py}` (whichever shape fits the framework) so a future reforge can find them in one place.

## When to deviate

The user owns the worker. If they say "I want it lighter" or "make it blue not blurple," do that. The default exists so the *uninstructed* case still looks good — it's not a religion. Record any deviation in `AUTHORING.md` so the next reforge doesn't quietly walk it back to defaults.
